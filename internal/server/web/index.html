Guide

Everything you need to know about dck — installation, configuration, commands, and best practices.

What is dck?

dck is a lightweight container runtime written in Go. It provides Docker-compatible commands for managing containers with minimal overhead. dck-client is the web management UI that connects to dck.

Installation

Install dck runtime:
curl -sSL https://raw.githubusercontent.com/animesao/dck/main/install.sh | sudo bash
Install dck-client (web UI):
curl -sSL https://raw.githubusercontent.com/animesao/dck-client/main/install.sh | sudo bash
Update dck-client:
dck-client update

The install script builds from source, so it requires Go to be installed on your system.

Configuration

dck-client can be configured through the web UI or by editing the settings file directly.

dck-client config file location:
/etc/dck-client/config.toml
Key settings:
dck_binary_pathPath to the dck binary (default: dck)
dck_data_dirdck data directory (default: /var/lib/dck)
registration_openAllow new user registration via the web UI

dck.toml — Stack Config

Define multi-container stacks using dck.toml. Similar to Docker Compose but native to dck.

# /etc/dck/dck.toml
[containers.web]
  image = "nginx:alpine"
  ports = ["80:80"]
  restart = "always"

[containers.db]
  image = "postgres:16-alpine"
  ports = ["5432:5432"]
  env = ["POSTGRES_PASSWORD=secret"]
  volumes = ["pg_data:/var/lib/postgresql/data"]

Use the Config tab in the web UI to edit, deploy, and stop stacks defined in dck.toml.

Containers — Basic Commands

dck run nginx:alpineRun a container
dck psList running containers
dck ps -aList all containers
dck stop <name>Stop a container
dck start <name>Start a stopped container
dck rm <name>Remove a container
dck logs <name>View container logs
dck exec <name> shEnter a running container

Images

dck pull nginx:alpinePull an image from registry
dck imagesList downloaded images
dck rmi nginx:alpineRemove an image

Images are pulled automatically when you create a container. You can also pre-pull images from the Images tab.

Port Mapping

Expose container ports to the host. Format: host_port:container_port.

# Map host port 8080 to container port 80
dck run -p 8080:80 nginx:alpine

# Map multiple ports
dck run -p 80:80 -p 443:443 nginx:alpine

# Random host port
dck run -p 80 nginx:alpine

Volumes

Persist data and share files between host and container. Format: host_path:container_path.

# Bind mount a host directory
dck run -v /data/mysql:/var/lib/mysql mysql:8

# Named volume (managed by dck)
dck run -v mydata:/data alpine

# Multiple volumes
dck run -v /config:/etc/config -v /data:/var/data app
Named volumes persist even after container removal. Bind mounts depend on the host path existing.

Environment Variables

Pass environment variables to containers for configuration.

# Single variable
dck run -e NODE_ENV=production node:22-alpine

# Multiple variables
dck run -e DB_HOST=db -e DB_PORT=5432 -e DB_NAME=myapp postgres:16-alpine

# From file
dck run --env-file .env app

Resource Limits

Control how much CPU and memory a container can use.

--memory 512mLimit memory to 512 megabytes
--memory 2gLimit memory to 2 gigabytes
--cpus 1.5Limit to 1.5 CPU cores
--cpus 0.5Limit to half a CPU core
dck run --memory 512m --cpus 1.0 nginx:alpine

Restart Policies

noDo not automatically restart (default)
alwaysAlways restart if stopped
on-failureRestart only if container exits with error
unless-stoppedAlways restart unless explicitly stopped
dck run --restart always nginx:alpine

Blueprints

Blueprints are pre-configured templates for popular applications. They provide sensible defaults for image, ports, environment variables, and volumes.

To use a blueprint:

  1. Go to the Blueprints tab
  2. Select a template (Nginx, PostgreSQL, Redis, etc.)
  3. Customize the name, ports, env vars, and other settings
  4. Click Pull & Deploy to launch

Best Practices

  • Use specific tags — avoid latest, pin versions like nginx:1.25
  • Set resource limits — always set --memory and --cpus for production
  • Use named volumes — for persistent data instead of bind mounts
  • Least privilege — run containers as non-root user when possible
  • Health checks — define healthcheck commands for your containers
  • Use dck.toml — for multi-container stacks and repeatable deployments
